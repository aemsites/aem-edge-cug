# AGENTS.md

## Project Overview

Closed User Groups (CUG) for AEM Edge Delivery Services — a multi-project repository containing an Edge Delivery site and CDN edge workers (Akamai and Cloudflare) that enforce spreadsheet-driven, per-path access control via Adobe IMS OAuth 2.0 + PKCE. Authors define which paths to protect and which groups are allowed; the CDN worker authenticates visitors and checks group membership before serving content.

This is an example implementation, not an official reference. Adapt the patterns to your own site.

## Architecture

```
Author spreadsheet (closed-user-groups)
    │
    ▼
AEM Config Service
    │  sets x-aem-cug-required / x-aem-cug-groups headers
    ▼
CDN Edge Worker (Akamai or Cloudflare)
    │  1. Proxy request to AEM origin
    │  2. Read CUG headers from origin response
    │  3. If x-aem-cug-required=true and no valid session → OAuth redirect to Adobe IMS
    │  4. On callback: exchange code for tokens (PKCE), create signed JWT session cookie
    │  5. Check x-aem-cug-groups against session groups
    │  6. Strip CUG headers, serve page (or 403)
    ▼
Browser (receives page only if authorized)
```

Sessions are stateless signed JWTs in an `auth_token` cookie (HMAC-SHA256, 1-hour TTL). PKCE state is stored in Cloudflare KV (5-min TTL) or encoded in the OAuth state parameter as a signed JWT (Akamai).

## Deployable Concerns

This repo contains four independent deployable things. They share documentation but have separate dependency management and deploy processes.

### 1. Edge Delivery Site (root)

The AEM Edge Delivery site — blocks, scripts, styles, and component models. Deployed automatically via AEM Code Sync when pushed to GitHub.

```bash
npm ci                # install dev dependencies (linters, model merge)
npm run lint          # ESLint (airbnb) + Stylelint
aem up                # local dev server at http://localhost:3000
```

### 2. Cloudflare Worker: cug-cloudflare-worker

Generic CUG worker using Adobe IMS OAuth + PKCE. Uses Cloudflare KV for PKCE state.

```bash
cd workers/cloudflare/cug-cloudflare-worker
npm ci
npm test              # Vitest
npm run dev           # local dev via wrangler
npm run deploy        # deploy to Cloudflare
npm run deploy -- --env summit   # deploy to summit environment
```

### 3. Cloudflare Worker: frescopa-b2b-worker

Frescopa B2B variant — uses a local user store (`users.js`) and POST-based login instead of IMS OAuth.

```bash
cd workers/cloudflare/frescopa-b2b-worker
npm ci
npm test              # Vitest
npm run dev
npm run deploy
```

### 4. Akamai EdgeWorker: cug-akamai-worker

Akamai EdgeWorker using Adobe IMS OAuth + PKCE. Secrets are bundled in `secrets.js` (gitignored). PKCE state is encoded in the OAuth state parameter as a signed JWT (no EdgeKV needed).

```bash
cd workers/akamai/cug-akamai-worker
cp secrets.example.js secrets.js    # fill in your values
tar -czvf bundle.tgz bundle.json main.js config.js secrets.js oauth.js session.js cug.js utils.js
akamai edgeworkers upload --bundle bundle.tgz <edgeworker-id>
akamai edgeworkers activate <edgeworker-id> STAGING 1.0
```

No npm `package.json` — the Akamai worker uses Akamai's built-in module system.

## Conventions

- **JavaScript:** ES6+, Airbnb ESLint rules, `.js` file extensions required in imports, Unix line endings
- **CSS:** Stylelint standard config, 4-space indent
- **No build step:** The EDS site and Cloudflare workers use vanilla JS with no transpilation
- **Husky pre-commit:** Editing any `models/_*.json` file auto-runs `npm run build:json` to regenerate the merged `component-*.json` files
- **Workers are independent:** Each worker has its own `package.json` and deploys separately. The root `package.json` does not use npm workspaces
- **Secrets are never committed:** Akamai uses `secrets.js` (gitignored); Cloudflare uses `wrangler secret put`

## Key Patterns

### CUG Header Enforcement

All workers share the same enforcement pattern in `cug.js`:

1. Read `x-aem-cug-required` from origin response — if `true`, authentication is required
2. Read `x-aem-cug-groups` — if present, the user must belong to at least one listed group
3. Strip both headers before forwarding the response to the browser
4. Set `Cache-Control: private, no-store` on protected pages

### Group Derivation

By default, groups are derived from email domain (`user@adobe.com` → group `adobe.com`). This can be adapted in `oauth.js` (Cloudflare/Akamai) to use IdP claims, external lookups, or custom mapping.

### Auth Endpoints

All workers expose:

| Endpoint | Purpose |
|----------|---------|
| `/auth/me` | Returns `{ authenticated, email, name, groups }` or `{ authenticated: false }` |
| `/auth/login` | Starts OAuth flow (or POST login for frescopa-b2b) |
| `/auth/callback` | OAuth code exchange (not used by frescopa-b2b) |
| `/auth/logout` | Clears session, redirects to IMS logout |
| `/auth/portal` | Redirects to group-specific path based on mapping sheet |

### Client-Side Auth (Header Block)

`blocks/header/header.js` calls `GET /auth/me` on page load. If authenticated, it renders a user dropdown with sign-out. If not, it renders a "Sign in" link pointing to `/auth/login?redirect=<current path>`.

### User Group Teaser Block

`blocks/user-group-teaser/user-group-teaser.js` calls `/auth/me`, matches user groups against a fragment map, and loads a group-specific teaser. This is a UX feature — the actual security boundary is at the CDN edge.

## Constraints

- **No npm workspaces** — the root and each worker manage dependencies independently
- **Akamai httpRequest limitation** — EdgeWorker `httpRequest` can only reach hostnames served by the same Akamai property. IMS token exchange must be proxied through a Property Manager rule (`/ims/token/v3` → `ims-na1.adobelogin.com`)
- **Cloudflare KV** — used only for PKCE state (5-min TTL), not for sessions
- **`.hlxignore` excludes `*.md`** — Markdown files (including this one) are not served by Edge Delivery. This is intentional; these files are for repo-level consumption only
